## Phase 1: Refactor (подготовка к библиотеке)

Цель — стабильный public API, готовый к расширению без переписывания.

### 1.1 Унификация ArbiterQueue

- Удалить ArbiterQueue из aether-mcp
- Единственный ArbiterQueue — в aether-core
- Обогатить UUID-полем из mcp-версии если нужно
- Все потребители (TUI, Web, MCP) используют один Arc<Mutex<ArbiterQueue>>

### 1.2 Event system

- Решить судьбу DiagnosticEvent: либо убрать, либо сделать центральным
- Определить trait EventBus в core — подготовка к gRPC streaming
- Все inter-crate коммуникации через typed events, не raw Vec

### 1.3 Public API surface

- Ревью каждого crate: pub vs pub(crate)
- lib.rs — только pub mod + pub use re-exports
- Все pub items — /// doc-comments
- #[non_exhaustive] на public enums которые будут расширяться (DiagnosticSeverity, ProbeStatus, etc.)

### 1.4 Error handling

- Заменить все .expect() в web handlers на proper error return
- Каждый crate — свой Error enum через thiserror
- Consistent pattern: Result<T, CrateError> для fallible pub methods

### 1.5 Trait abstractions (подготовка к v1.0)

Определить в aether-core traits для будущих crate-ов, чтобы они подключались как адаптеры:

```
trait DataSource          — Prometheus scraper, prober, log parser
trait DiagnosticRule      — уже есть, проверить API
trait OutputSink          — Slack, Discord, Telegram, stdout, file
trait ServiceDiscovery    — port scan, K8s API, docker
trait ConfigProvider      — TOML, YAML, env
```

Эти traits определяют контракт. Новые crate-ы реализуют их. Существующий код не меняется.

### 1.6 SharedState pattern

- Единообразный подход: все shared state через Arc<RwLock<T>> для reads, Arc<Mutex<T>> для exclusive
- Diagnostic deduplication в SharedState, не в main.rs
- MetricStore — единое хранилище для всех источников данных

**Критерий завершения Phase 1:**
- Public API задокументирован
- Traits для будущих crate-ов определены в core
- 0 дублирования типов между crate-ами
- Error handling единообразен
- SharedState без race conditions

## Принцип аддитивности

Архитектура построена так, чтобы каждый Phase дополнял предыдущий:

```
Phase 0: Fix bugs in existing code
         ↓ (код стабилен)
Phase 1: Define traits + clean API
         ↓ (контракты заложены)
Phase 2: New crates implement traits
         ↓ (ни один существующий файл не переписан)
Phase 3: More implementations of same traits
         ↓ (ещё больше адаптеров)
Future:  Other projects consume Aether API
         (gRPC + event bus уже готов)
```

Каждый новый crate:
1. Зависит только от aether-core
2. Реализует trait из core
3. Подключается в aether-terminal/main.rs через dependency injection
4. Существующий код не трогается

Это гарантирует что Auto-Fix Agent, K8s Autoscaler и Service Graph смогут подключиться к Aether через тот же gRPC/Event Bus без рефакторинга.
